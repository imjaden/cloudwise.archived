## 技术栈

- 单文件 HTML（HTML + 内联 CSS + 内联 JS）
- 深色主题（`#0A0A2A` 深海蓝）
- 响应式设计，支持移动端
- Lightbox 图片放大查看
- 无外部依赖，零构建成本
- v1.2 版本采用数据驱动渲染（`TIMELINE_DATA` + `renderTimeline()`），提升可维护性

## 附录：index.html vs index-v1.2.html 前端对比分析

### 1. 传输体积

| 指标 | index.html | index-v1.2 | 差异 |
|:---|:---|:---|:---|
| 原始体积 | 45,043 B | 36,517 B | **-19%** |
| Gzip 体积 | 10,013 B | 10,417 B | +4% |
| HTML 元素数 | 272 | 63 | **-77%** |
| 内联 JS | 2.9 KB | 17.5 KB | +507% |

- 原始体积 v1.2 小 19%，但 Gzip 后反而大 4%。原因：HTML 标签重复模式 gzip 压缩率极高（~78%），JS 对象字面量压缩率较低（~40%）
- 实际网络传输（开启 Gzip）两者几乎持平，v1.2 略输

### 2. 渲染性能（Critical Rendering Path）

| 阶段 | index.html | index-v1.2 |
|:---|:---|:---|
| HTML 解析 | 一次完成，272 个节点 | 解析出 63 个空容器 |
| CSS 计算 | 同（144 行，完全一致） | 同 |
| JS 执行 | Param 脚本（微秒级） | Param + 数据解析 + 2 次 render（~1-3ms） |
| 首屏内容 | 立即可见 | 空容器 → JS 填充（可能感知闪烁） |

- index.html：纯静态，浏览器解析完 HTML 即可绘制，零 JS 阻塞
- v1.2：需要等 JS 执行完 timeline + contrib 才能绘制完整内容
  - 在低端设备 / 慢速 CPU 上，15 项 timeline 的 innerHTML 拼接 + DOM 插入可能产生 50-200ms 延迟
  - IntersectionObserver 动画在 JS 渲染后才能 observe 到元素
- 结论：首屏渲染 index.html 更快，差距在高端设备可忽略

### 3. 图片加载策略

| | index.html | index-v1.2 |
|:---|:---|:---|
| 静态 HTML 中 | 19 张 img 标签 | 2 张（profile + portrait） |
| JS 动态插入 | 0 | 18 张 |
| loading="lazy" | 全部使用 | 全部使用 |

- 两者都用了 `loading="lazy"`，浏览器预扫描机制不同：
  - index.html：浏览器解析 HTML 时就能发现所有 img 标签，可以提前建立连接、预分配解码缓冲区
  - v1.2：18 张图片由 JS 动态插入，浏览器无法在 HTML 预扫描中发现它们，图片发现时机延迟到 JS 执行后
- 实际影响很小（图片都在首屏以下，lazy 本身会延迟加载）

### 4. SEO & 社交分享

| | index.html | index-v1.2 |
|:---|:---|:---|
| 搜索引擎爬虫 | 看到完整 15 项内容 | 看到空容器 + JS 代码 |
| OG/Twitter meta | 正常（head 中） | 正常（head 中） |
| 结构化数据 | 可直接提取 | 需要执行 JS |

- 如果页面仅在 GitHub Pages 静态托管，Google 爬虫可以执行 JS，但百度爬虫不一定。对于个人档案页影响不大，但如果需要 SEO，index.html 的纯静态方案更稳妥

### 5. 可维护性（v1.2 的核心优势）

| 操作 | index.html | index-v1.2 |
|:---|:---|:---|
| 新增 1 条荣誉 | 复制 20 行 HTML 模板 | 追加 1 个 JS 对象 |
| 修改描述文字 | 定位到具体 HTML 行 | 改 desc 字段 |
| 调整顺序 | 剪切粘贴 HTML 块 | 改数组顺序 |
| 批量改样式 | CSS + 每个 HTML 块 | CSS + render 函数 |
| 数据复用 | 不可能 | TIMELINE_DATA 可被搜索/筛选/导出 |

- v1.2 的维护成本显著低于 index.html
- 15 项荣誉 → 每次改动省 5-10 分钟，且不容易出错

### 6. 运行时体验细节

| | index.html | index-v1.2 |
|:---|:---|:---|
| innerHTML 拼接 | 无 | 15 次字符串拼接 |
| DOM 操作 | 0 | appendChild × 15 |
| DocumentFragment | N/A | 已使用（减少重排） |
| XSS 风险 | 无（静态） | desc/title 中如含 `<script>` 会被执行（当前数据安全） |

- v1.2 用了 DocumentFragment 批量插入，DOM 操作已优化
- 但 innerHTML 字符串拼接有 XSS 隐患（如果数据来源不可信）

### 7. 综合结论

| 维度 | 胜出方案 | 差距 |
|:---|:---|:---|
| 首屏速度 | index.html | 微弱 |
| Gzip 体积 | index.html | 微弱（4%） |
| SEO 友好度 | index.html | 中等 |
| 可维护性 | **index-v1.2** | **显著** |
| 代码复用 | **index-v1.2** | **显著** |
| 数据可扩展性 | **index-v1.2** | **显著** |
| 运行时性能 | index.html | 微弱 |

**推荐**：日常维护用 v1.2，发布到生产时可考虑 SSR 或构建时将数据预渲染回静态 HTML（两全其美）。
